
1. Полная иерархия инициализации
Когда создается объект Derived, C++ следует строгому алгоритму. чтобы собрать внешнюю куклу, нужно сначала собрать все внутренние.
Порядок создания (Constructor Order):
 * Базовый класс (Base):
   * Сначала инициализируются поля базового класса (объект A a).
   * Затем выполняется тело конструктора Base.
 * Производный класс (Derived):
  * Инициализируются собственные поля наследника (объект B b).
   * Выполняется тело конструктора Derived.
2. Развернутый пример с композицией
```cpp
#include <iostream>
#include <string>

// Вспомогательный класс для демонстрации полей
struct A {
    A() { std::cout << "  [A] Constructed (field of Base)\n"; }
    ~A() { std::cout << "  [A] Destroyed\n"; }
};

struct B {
    B() { std::cout << "  [B] Constructed (field of Derived)\n"; }
    ~B() { std::cout << "  [B] Destroyed\n"; }
};

// Базовый класс

struct Base {
    A a;        // Сначала создастся это поле
    int x = 0;

    Base() { std::cout << " [Base] Default Constructor\n"; }
    
    Base(int x_val) : x(x_val) { 
        std::cout << " [Base] Parameterized Constructor (x=" << x << ")\n"; 
    }

    ~Base() { std::cout << " [Base] Destructor\n"; }
};

// Производный класс

struct Derived : Base {
    B b;        // Это поле создастся после того, как Base будет готов

    // Явный вызов конструктора родителя через список инициализации
    
    Derived() : Base(5) { 
        std::cout << "[Derived] Constructor\n"; 
    }

    ~Derived() { std::cout << "[Derived] Destructor\n"; }
};

int main() {
    std::cout << "--- Start ---\n";
    {
        Derived d;
    }
    std::cout << "--- End ---\n";
    return 0;
}
```

Результат в консоли (трассировка):
 * [ ] Constructed — пошло создание Base, сначала его поля.
 * [Base] Parameterized Constructor (x=5) — отработал сам Base.
 * [ ] Constructed — пошло создание Derived, сначала его поля.
 * [Derived] Constructor — отработал сам Derived.
 * [Derived] Destructor — начали удалять: сначала тело наследника.
 * [ ] Destroyed — удалили поле наследника.
 * [Base] Destructor — удалили тело базы.
 * [ ] Destroyed — удалили поле базы.


3. Зачем это нужно? (Теория для экзамена)
Список инициализации (Initialization List)
На фото в строке Derived() : Base(5) используется двоеточие. Это критически важный момент.
 * Если убрать : Base(5), компилятор автоматически подставит вызов Base().
 * Если у Base нет конструктора без параметров, программа выдаст ошибку.
 * Правило: Мы используем список инициализации, чтобы передать данные наверх «родителю».
Почему деструкторы работают наоборот?
Представь, что класс Derived использует переменную x из класса Base. Если мы сначала удалим Base, то в деструкторе Derived мы обратимся к уже несуществующей памяти. Поэтому C++ сначала «снимает крышу» (наследника), а потом «разбирает фундамент» (базу).
Проблема «Срезания» (Object Slicing) — на будущее
Это часто спрашивают на зачетах. Если вы попытаетесь присвоить объект Derived переменной типа Base (не по ссылке), то вся «добавка» от Derived (поле B b) будет отрезана, и останется только чистый Base.
Тебе нужно разобрать, как работают виртуальные функции в этой связке, или пока достаточно порядка конструкторов?

## Приведение типов, срезка (slicing) и виды наследования
#### Приведение типов в иерархиях наследования

В C++ существуют три основных типа преобразований между базовыми и производными классами:

1. **Upcast (приведение к базовому типу)**
   - Преобразование указателя/ссылки наследника к указателю/ссылке родителя
   - Выполняется **неявно** и **безопасно**
   - Позволяет работать с наследником через интерфейс базового класса

2. **Downcast (приведение к производному типу)**
   - Преобразование указателя/ссылки родителя к указателю/ссылке наследника
   - Выполняется **явно** и **потенциально опасно**
   - Используется `static_cast` (если тип известен) или `dynamic_cast` (с проверкой)

3. **Срезка (Slicing)**
   - Возникает при копировании объекта наследника в объект базового класса
   - Данные, специфичные для наследника, "отрезаются" и теряются
   - Чтобы избежать срезки, используют указатели или ссылки

#### Виды наследования

| Тип наследования | public члены родителя | protected члены родителя | private члены родителя |
|------------------|----------------------|-------------------------|-----------------------|
| **public** | public | protected | недоступны |
| **protected** | protected | protected | недоступны |
| **private** | private | private | недоступны |

**Особенности:**
- **public наследование**: сохраняет отношение "is-a" (наследник является разновидностью родителя)
- **protected/private наследование**: используется для реализации "implemented-in-terms-of" (наследник реализован с помощью родителя)
- При private/protected наследовании неявное приведение к базовому классу доступно **только внутри** класса-наследника

```cpp
        std::cout << "Base destroyed\n"; 
    }

struct Derived : public Base {

    std::string name = "derived";

    }

    }

void g(Derived& d) {
    std::cout << "g(Derived&): derived_data = " << d.derived_data << "\n";
```

#### 3. Срезка (Slicing)

```cpp
int main() {
    std::cout << "=== Slicing Examples ===\n\n";
    
    Derived d(5, 20);
    std::cout << "Original Derived: a=" << d.a 
              << ", derived_data=" << d.derived_data << "\n\n";
    
    // Срезка при копировании в объект базового класса
    Base b = d;  // Вызывается Base(const Base&) - derived_data теряется
    std::cout << "After slicing: a=" << b.a << "\n";
    // b.derived_data; // Ошибка! Поле недоступно
    
    // Срезка при передаче по значению
    std::cout << "\nPassing by value:\n";
    auto passByValue = [](Base b) {
        std::cout << "Inside function: a=" << b.a << "\n";
    };
    passByValue(d);  // Снова срезка
    
    // НО! Срезки нет при работе с указателями и ссылками
    std::cout << "\nPassing by reference:\n";
    auto passByReference = [](Base& b) {
        std::cout << "Inside function: a=" << b.a << "\n";
    };
    passByReference(d);  // Нет срезки, работает с оригиналом
    
    return 0;
}
```
#### 4. Явное приведение типов

```cpp
int main() {
    std::cout << "=== Explicit Casting ===\n\n";
    
    Derived d(10, 30);
    Base& base_ref = d;
    
    // static_cast - безопасен, если мы знаем реальный тип
    std::cout << "1. static_cast to Base& (upcast):\n";
    Base& b = static_cast<Base&>(d);
    std::cout << "   b.a = " << b.a << "\n\n";
    
    // static_cast для значения - вызывает срезку
    std::cout << "2. static_cast to Base (slicing):\n";
    Base b_copy = static_cast<Base>(d);
    std::cout << "   b_copy.a = " << b_copy.a << "\n\n";
    
    // static_cast downcast - требует уверенности
    std::cout << "3. static_cast to Derived& (downcast):\n";
    Derived& d_ref = static_cast<Derived&>(base_ref);
    std::cout << "   d_ref.derived_data = " << d_ref.derived_data << "\n\n";
    
    // reinterpret_cast - опасно, просто переинтерпретирует биты
    std::cout << "4. reinterpret_cast (dangerous):\n";
    int* int_ptr = reinterpret_cast<int*>(&d);
    std::cout << "   First bytes as int: " << *int_ptr << "\n";
    // ВНИМАНИЕ! reinterpret_cast обходит систему типов
    
    return 0;
}
```

#### 5. Private и Protected наследование

```cpp
// Демонстрация разных типов наследования
class PublicBase {
public:
    int pub = 1;
protected:
    int prot = 2;
private:
    int priv = 3;
};

// Public наследование
class PublicDerived : public PublicBase {
public:
    void show() {
        std::cout << pub << " " << prot << std::endl;  // priv недоступен
    }
};

// Protected наследование
class ProtectedDerived : protected PublicBase {
public:
    void show() {
        std::cout << pub << " " << prot << std::endl;  // priv недоступен
    }
    // pub и prot становятся protected в этом классе
};

// Private наследование
class PrivateDerived : private PublicBase {
public:
    void show() {
        std::cout << pub << " " << prot << std::endl;  // priv недоступен
    }
    // pub и prot становятся private в этом классе
};

int main() {
    std::cout << "=== Access Types in Inheritance ===\n\n";
    
    PublicDerived pub_d;
    pub_d.pub = 10;     // OK: pub доступен снаружи
    // pub_d.prot = 20; // Ошибка: prot защищен
    
    ProtectedDerived prot_d;
    // prot_d.pub = 10; // Ошибка: pub стал protected
    prot_d.show();      // OK: доступ внутри класса
    
    PrivateDerived priv_d;
    // priv_d.pub = 10;  // Ошибка: pub стал private
    priv_d.show();       // OK: доступ внутри класса
    
    // Важно: неявное приведение к базовому классу
    PublicBase* p1 = &pub_d;      // OK: public наследование
    
    // ProtectedBase* p2 = &prot_d; // Ошибка: нет неявного приведения
    ProtectedBase* p2 = (ProtectedBase*)&prot_d; // C-style cast (опасно)
    
    // PrivateBase* p3 = &priv_d;    // Ошибка: нет неявного приведения
    
    return 0;
}
```

### Ключевые выводы

1. **Upcast** безопасен и выполняется неявно
2. **Downcast** требует явного приведения и осторожности
3. **Срезка** происходит при копировании наследника в объект базового класса
4. Чтобы избежать срезки, используйте **указатели** или **ссылки**
5. **Public** наследование сохраняет отношение "is-a"
6. **Protected/private** наследование ограничивает видимость и приведение типов
7. Для полиморфного поведения используйте **виртуальные функции** и передавайте объекты по ссылке/указателю

#### Множественное наследование

Множественное наследование — это возможность класса наследоваться от двух или более базовых классов. Это позволяет комбинировать функциональность из разных источников.

**Синтаксис:**
```cpp
class Derived : public Base1, public Base2, private Base3 {
    // ...
};
```

**Преимущества:**
- Позволяет создавать классы, комбинирующие функциональность нескольких независимых иерархий
- Реализует концепцию "примесей" (mixins)
- Полезен при проектировании интерфейсов

**Недостатки:**
- Усложняет понимание кода
- Проблема ромба (diamond problem)
- Неоднозначность при обращении к членам с одинаковыми именами
- Увеличение сложности управления памятью

#### Проблема ромба (Diamond Problem)

Проблема возникает, когда два класса наследуют одного общего предка, а затем третий класс наследует от обоих этих классов.

**Структура ромба:**
```
      GrandBase
       /    \
      /      \
   Base1    Base2
      \      /
       \    /
        Derived
```

**Проблемы:**
1. **Дублирование данных**: В объекте `Derived` содержится **две копии** `GrandBase`
2. **Неоднозначность доступа**: При обращении к члену `GrandBase` непонятно, к какой копии обращаться
3. **Избыточность**: Дублирование данных приводит к неэффективному использованию памяти
4. **Сложность инициализации**: Конструктор `GrandBase` вызывается дважды

#### Виртуальное наследование (Virtual Inheritance)

**Решение проблемы ромба** — использование виртуального наследования с ключевым словом `virtual`.

**Как это работает:**
- Виртуальное наследование гарантирует, что в иерархии существует **только один экземпляр** виртуального базового класса
- Компилятор добавляет механизм (обычно указатели) для доступа к общему экземпляру
- Порядок инициализации изменяется: сначала инициализируются виртуальные базовые классы

**Синтаксис:**
```cpp
class Base1 : virtual public GrandBase { };
class Base2 : virtual public GrandBase { };
class Derived : public Base1, public Base2 { };
```

```cpp
        std::cout << "DiamondProblem constructor with params\n";
    }
    
    void show() const override {
        std::cout << "DiamondProblem:\n";
        std::cout << "  From BaseA: ";
        BaseA::show();
        std::cout << "  From BaseB: ";
        BaseB::show();
        std::cout << "  derived_data=" << derived_data << "\n";

    // Проблема: как обратиться к GrandBase::value?
    void demonstrateAmbiguity() {
        // BaseA::value и BaseB::value - разные копии!
        BaseA::setValue(42);    // Изменяет только копию в BaseA
        BaseB::setValue(99);    // Изменяет только копию в BaseB
        
        std::cout << "\nAfter modifications:\n";
        std::cout << "BaseA::value = " << BaseA::getValue() << "\n";
        std::cout << "BaseB::value = " << BaseB::getValue() << "\n";
        
        // Нет единого value!
    }
    
    ~DiamondProblem() override {
        std::cout << "DiamondProblem destructor\n";
    }
};

int main() {

    obj.demonstrateAmbiguity();
    std::cout << "\n";
    
    std::cout << "Size of DiamondProblem: " << sizeof(DiamondProblem) << " bytes\n";

    return 0;
```

```cpp
class VirtualGrandBase {
protected:
    int value;
    
public:
    VirtualGrandBase() : value(0) {
        std::cout << "VirtualGrandBase constructor, value=" << value << "\n";

    }

        std::cout << "VirtualGrandBase::setValue(" << v << ")\n";
    }

    }

        std::cout << "VirtualGrandBase: value=" << value << "\n";
    }
    
    virtual ~VirtualGrandBase() {
        std::cout << "VirtualGrandBase destructor\n";
    }
};

// ВИРТУАЛЬНОЕ наследование от GrandBase
class VirtualBaseA : virtual public VirtualGrandBase {
protected:
    int a_data;

    // Важно: конструктор VirtualGrandBase вызывается из самого производного класса!
    VirtualBaseA() : VirtualGrandBase(), a_data(100) {
        std::cout << "VirtualBaseA constructor, a_data=" << a_data << "\n";
    }
    
    VirtualBaseA(int v, int a) : VirtualGrandBase(v), a_data(a) {

    }
    
    void show() const override {
        std::cout << "VirtualBaseA: value=" << getValue() << ", a_data=" << a_data << "\n";
    }
    
    ~VirtualBaseA() override {
        std::cout << "VirtualBaseA destructor\n";
    }
};

// ВИРТУАЛЬНОЕ наследование от GrandBase
class VirtualBaseB : virtual public VirtualGrandBase {
protected:
    int b_data;
    
public:
    VirtualBaseB() : VirtualGrandBase(), b_data(200) {
        std::cout << "VirtualBaseB constructor, b_data=" << b_data << "\n";
    }
    
    VirtualBaseB(int v, int b) : VirtualGrandBase(v), b_data(b) {
        std::cout << "VirtualBaseB constructor with params, b_data=" << b_data << "\n";
    }
    
    void show() const override {
        std::cout << "VirtualBaseB: value=" << getValue() << ", b_data=" << b_data << "\n";
    }
    
    ~VirtualBaseB() override {
        std::cout << "VirtualBaseB destructor\n";
    }
};

// Решение проблемы ромба
class DiamondSolution : public VirtualBaseA, public VirtualBaseB {
    int derived_data;
    
public:
    // ВАЖНО: самый производный класс инициализирует виртуальный базовый класс
    DiamondSolution() 
        : VirtualGrandBase(0),  // Инициализация виртуального базового
          VirtualBaseA(), 
          VirtualBaseB(), 
          derived_data(300) {
        std::cout << "DiamondSolution constructor, derived_data=" << derived_data << "\n";
    }
    
    DiamondSolution(int v, int a, int b, int d) 
        : VirtualGrandBase(v),  // Один вызов конструктора VirtualGrandBase
          VirtualBaseA(v, a),   // Передаем v, но он будет проигнорирован
          VirtualBaseB(v, b),   // Передаем v, но он будет проигнорирован
          derived_data(d) {
        std::cout << "DiamondSolution constructor with params\n";
        // VirtualGrandBase инициализируется только один раз!
    }
    
    void show() const override {
        std::cout << "DiamondSolution:\n";
        std::cout << "  VirtualGrandBase: ";
        VirtualGrandBase::show();
        std::cout << "  VirtualBaseA: ";
        VirtualBaseA::show();
        std::cout << "  VirtualBaseB: ";
        VirtualBaseB::show();
        std::cout << "  derived_data=" << derived_data << "\n";
    }
    
    void demonstrateCorrectBehavior() {
        // Теперь есть только ОДНА копия VirtualGrandBase
        setValue(42);  // Изменяет единое значение
        std::cout << "After setValue(42):\n";
        std::cout << "VirtualBaseA::getValue() = " << VirtualBaseA::getValue() << "\n";
        std::cout << "VirtualBaseB::getValue() = " << VirtualBaseB::getValue() << "\n";
        
        VirtualBaseA::setValue(99);  // Все равно меняет то же самое
        std::cout << "After VirtualBaseA::setValue(99):\n";
        std::cout << "getValue() = " << getValue() << "\n";
    }
    
    ~DiamondSolution() override {
        std::cout << "DiamondSolution destructor\n";
    }
};

int main() {
    std::cout << "=== Diamond Problem Solution (virtual inheritance) ===\n\n";
    
    DiamondSolution obj(10, 1000, 2000, 3000);
    std::cout << "\n";
    obj.show();
    std::cout << "\n";
    obj.demonstrateCorrectBehavior();
    std::cout << "\n";
    
    std::cout << "Size of DiamondSolution: " << sizeof(DiamondSolution) << " bytes\n";
    // Объект содержит ОДНУ копию VirtualGrandBase + указатели для доступа
    
    return 0;
}
```

```cpp
// Демонстрация порядка вызова конструкторов
class Root {
public:
    Root() { std::cout << "Root\n"; }
    Root(int) { std::cout << "Root(int)\n"; }
};

class Base1 : virtual public Root {
public:
    Base1() { std::cout << "Base1\n"; }
    Base1(int x) : Root(x) { std::cout << "Base1(int)\n"; }
};

class Base2 : virtual public Root {
public:
    Base2() { std::cout << "Base2\n"; }
    Base2(int x) : Root(x) { std::cout << "Base2(int)\n"; }
};

class Derived1 : public Base1, public Base2 {
public:
    // Root будет инициализирован здесь (по умолчанию)
    Derived1() { std::cout << "Derived1\n"; }
    
    // Можно явно указать, как инициализировать Root
    Derived1(int x) : Root(x), Base1(x), Base2(x) {
        std::cout << "Derived1(int)\n";
    }
};

class Derived2 : virtual public Base1, virtual public Base2 {
public:
    // Даже если Base1 и Base2 виртуальные, Root инициализируется в Derived2
    Derived2() { std::cout << "Derived2\n"; }
    
    Derived2(int x) : Root(x), Base1(x), Base2(x) {
        std::cout << "Derived2(int)\n";
    }
};

int main() {
    std::cout << "=== Constructor Order with Virtual Inheritance ===\n\n";
    
    std::cout << "1. Derived1 (without virtual):\n";
    Derived1 d1;
    // Root вызывается дважды! (через Base1 и Base2)
    
    std::cout << "\n2. Derived1 with parameter:\n";
    Derived1 d1_param(42);
    
    std::cout << "\n3. Derived2 (with virtual):\n";
    Derived2 d2;
    // Root вызывается только один раз!
    
    std::cout << "\n4. Derived2 with parameter:\n";
    Derived2 d2_param(100);
    
    return 0;
}
```

#### 5. Практический пример: интерфейсы с множественным наследованием

```cpp
// Интерфейсы (абстрактные классы)
class IDrawable {
public:
    virtual void draw() const = 0;
    virtual ~IDrawable() = default;
};

class IMovable {
public:
    virtual void move(double dx, double dy) = 0;
    virtual ~IMovable() = default;
};

class IClickable {
public:
    virtual void onClick(double x, double y) = 0;
    virtual bool isPointInside(double x, double y) const = 0;
    virtual ~IClickable() = default;
};

// Конкретная реализация
class Button : public IDrawable, public IClickable {
    std::string text;
    double x, y, width, height;
    
public:
    Button(const std::string& t, double x0, double y0, double w, double h)
        : text(t), x(x0), y(y0), width(w), height(h) {}
    
    void draw() const override {
        std::cout << "Drawing button '" << text << "' at (" << x << "," << y 
                  << ") size " << width << "x" << height << "\n";
    }
    
    void onClick(double clickX, double clickY) override {
        if (isPointInside(clickX, clickY)) {
            std::cout << "Button '" << text << "' clicked!\n";
        }
    }
    
    bool isPointInside(double px, double py) const override {
        return (px >= x && px <= x + width && py >= y && py <= y + height);
    }
};

// Класс, реализующий несколько интерфейсов
class DraggableButton : public Button, public IMovable {
    bool isDragging = false;
    
public:
    DraggableButton(const std::string& t, double x0, double y0, double w, double h)
        : Button(t, x0, y0, w, h) {}
    
    void move(double dx, double dy) override {
        // В реальном коде нужно было бы обновить позицию
        std::cout << "Moving button by (" << dx << "," << dy << ")\n";
    }
    
    void onClick(double clickX, double clickY) override {
        if (isPointInside(clickX, clickY)) {
            isDragging = true;
            std::cout << "Started dragging button\n";
        }
    }
};

int main() {
    std::cout << "=== Multiple Inheritance with Interfaces ===\n\n";
    
    Button okButton("OK", 10, 10, 80, 30);
    DraggableButton dragButton("Drag Me", 100, 100, 100, 40);
    
    // Использование интерфейсов
    std::vector<IDrawable*> drawables;
    drawables.push_back(&okButton);
    drawables.push_back(&dragButton);
    
    std::cout << "Drawing all drawables:\n";
    for (auto* d : drawables) {
        d->draw();
    }
    
    std::cout << "\nClicking on buttons:\n";
    okButton.onClick(15, 15);      // Внутри кнопки
    okButton.onClick(200, 200);    // Вне кнопки
    
    dragButton.onClick(110, 110);  // Начинаем перетаскивание
    
    // Использование интерфейса IMovable
    IMovable* movable = &dragButton;
    movable->move(10, 5);
    
    return 0;
}
```


1. **Множественное наследование** позволяет классу наследовать функциональность от нескольких базовых классов

2. **Проблема ромба** возникает, когда два класса наследуют одного общего предка, а третий класс наследует от обоих:
   - В объекте создается **две копии** общего предка
   - Возникает **неоднозначность** при обращении к членам
   - Увеличивается **размер** объекта

3. **Виртуальное наследование** (`virtual`) решает проблему ромба:
   - Гарантирует наличие **единственного экземпляра** виртуального базового класса
   - Изменяет **порядок инициализации** (виртуальные базовые классы инициализируются первыми)
   - Самый производный класс **отвечает** за инициализацию виртуальных базовых классов

4. **Правила инициализации** при виртуальном наследовании:
   - Сначала инициализируются **виртуальные** базовые классы (в порядке объявления)
   - Затем инициализируются **невиртуальные** базовые классы
   - Затем инициализируются **поля-члены**
   - В конце вызывается **конструктор** производного класса

5. **Рекомендации:**
   - Используйте множественное наследование **осознанно**
   - Применяйте виртуальное наследование **только** для решения проблемы ромба
   - Рассмотрите альтернативы: композицию, интерфейсы, примеси (mixins)
   - Помните, что виртуальное наследование добавляет **накладные расходы** (обычно указатели vptr)


#### 8.1 Идея полиморфизма и виртуальных функций

**Полиморфизм** — это третий принцип ООП (после инкапсуляции и наследования), который позволяет объектам разных типов по-разному реагировать на вызов одного и того же метода.

**Типы полиморфизма:**
1. **Статический (ранний)**: Реализуется через перегрузку функций и шаблоны. Решение о том, какая функция будет вызвана, принимается **на этапе компиляции**.
2. **Динамический (поздний)**: Реализуется через **виртуальные функции**. Решение о том, какая функция будет вызвана, принимается **во время выполнения программы**.

**Виртуальные функции:**
- Объявляются с ключевым словом `virtual` в базовом классе
- Могут быть переопределены в производных классах
- При вызове через **ссылку** или **указатель** на базовый класс вызывается версия, соответствующая реальному типу объекта
- Класс, содержащий хотя бы одну виртуальную функцию, называется **полиморфным**

**Механизм работы:**
- Для каждого полиморфного класса компилятор создает **таблицу виртуальных функций (vtable)**
- Каждый объект содержит скрытый указатель (**vptr**) на эту таблицу
- Вызов виртуальной функции происходит через косвенную адресацию: объект → vptr → vtable → функция

#### 8.2 Выбор версии между виртуальными и не виртуальными методами

**Правила переопределения:**
- Сигнатуры родителя и наследника должны **полностью совпадать** (включая const-квалификатор и параметры)
- Если сигнатуры не совпадают, происходит **сокрытие имени**, а не переопределение

**Ключевые слова:**
- **`override`** (C++11): Явно указывает, что метод переопределяет виртуальную функцию базового класса. Если сигнатура не совпадает — ошибка компиляции.
- **`final`** (C++11): Запрещает дальнейшее переопределение этого метода в наследниках.

**Важно:**
- Виртуальность функции **наследуется**: если функция объявлена `virtual` в базовом классе, она остается виртуальной во всех наследниках
- Статические функции **не могут** быть виртуальными
- Конструкторы **не могут** быть виртуальными
- Деструктор **должен** быть виртуальным в полиморфных классах

#### 8.3 Абстрактные классы и чисто виртуальные функции

**Чисто виртуальная функция** — это виртуальная функция, не имеющая реализации в базовом классе. Объявляется с `= 0`:
```cpp
virtual void draw() const = 0;
```

**Абстрактный класс:**
- Класс, содержащий хотя бы одну чисто виртуальную функцию
- **Нельзя создать объект** абстрактного класса
- Используется как **интерфейс** (контракт) для производных классов
- Производный класс **обязан** реализовать все чисто виртуальные функции, иначе он тоже станет абстрактным

#### 8.4 Проблема виртуального деструктора

**Проблема:**
Если базовый класс не имеет виртуального деструктора, то при удалении объекта производного класса через указатель на базовый класс:
- Вызывается **только** деструктор базового класса
- Деструктор производного класса **не вызывается**
- Ресурсы, выделенные в производном классе, **не освобождаются** → утечка памяти

**Решение:**
Всегда объявляйте деструктор **виртуальным** в классах, предназначенных для полиморфного использования.

#### 8.5 RTTI и dynamic_cast

**RTTI (Run-Time Type Information)** — механизм, позволяющий получать информацию о типе объекта во время выполнения.

**Операторы RTTI:**
1. **`typeid`**: Возвращает ссылку на `std::type_info`, содержащую информацию о типе
2. **`dynamic_cast`**: Безопасное приведение типов в иерархии с виртуальными функциями

**`dynamic_cast`:**
- Работает **только** с полиморфными типами (имеющими виртуальные функции)
- Для указателей: возвращает `nullptr` при неудачном приведении
- Для ссылок: кидает исключение `std::bad_cast` при неудачном приведении
- Требует **RTTI**, который можно отключить в настройках компилятора


```cpp
#include <iostream>
#include <vector>
#include <memory>

// Базовый класс
class Animal {
protected:
    std::string name;
    
public:
    Animal(const std::string& n) : name(n) {
        std::cout << "Animal constructor: " << name << "\n";
    }
    
    // Виртуальная функция
    virtual void speak() const {
        std::cout << name << " makes a sound\n";
    }
    
    // Не виртуальная функция
    void eat() const {
        std::cout << name << " is eating\n";
    }
    
    // Виртуальный деструктор обязателен!
    virtual ~Animal() {
        std::cout << "Animal destructor: " << name << "\n";
    }
};

// Производный класс
class Dog : public Animal {
private:
    std::string breed;
    
public:
    Dog(const std::string& n, const std::string& b) 
        : Animal(n), breed(b) {
        std::cout << "Dog constructor: " << breed << "\n";
    }
    
    // Переопределение виртуальной функции
    void speak() const override {
        std::cout << name << " says: Woof! Woof!\n";
    }
    
    // Специфичный для Dog метод
    void wagTail() const {
        std::cout << name << " is wagging tail\n";
    }
    
    ~Dog() override {
        std::cout << "Dog destructor: " << breed << "\n";
    }
};

class Cat : public Animal {
public:
    Cat(const std::string& n) : Animal(n) {
        std::cout << "Cat constructor\n";
    }
    
    void speak() const override {
        std::cout << name << " says: Meow!\n";
    }
    
    void purr() const {
        std::cout << name << " is purring\n";
    }
    
    ~Cat() override {
        std::cout << "Cat destructor\n";
    }
};

class Cow : public Animal {
public:
    Cow(const std::string& n) : Animal(n) {
        std::cout << "Cow constructor\n";
    }
    
    void speak() const override {
        std::cout << name << " says: Moo!\n";
    }
    
    ~Cow() override {
        std::cout << "Cow destructor\n";
    }
};

// Функция, демонстрирующая полиморфизм
void makeSound(const Animal& animal) {
    animal.speak();  // Вызов зависит от реального типа!
}

int main() {
    std::cout << "=== Polymorphism Demo ===\n\n";
    
    // 1. Прямой вызов
    Dog dog("Rex", "German Shepherd");
    Cat cat("Whiskers");
    Cow cow("Bessie");
    
    std::cout << "\nDirect calls:\n";
    dog.speak();  // Dog::speak
    cat.speak();  // Cat::speak
    cow.speak();  // Cow::speak
    
    // 2. Полиморфный вызов через ссылку
    std::cout << "\nPolymorphic calls through reference:\n";
    makeSound(dog);  // Вызовется Dog::speak
    makeSound(cat);  // Вызовется Cat::speak
    makeSound(cow);  // Вызовется Cow::speak
    
    // 3. Полиморфный вызов через указатель
    std::cout << "\nPolymorphic calls through pointer:\n";
    Animal* ptr = &dog;
    ptr->speak();  // Dog::speak
    ptr = &cat;
    ptr->speak();  // Cat::speak
    
    // 4. Важное отличие: виртуальные vs невиртуальные
    std::cout << "\nVirtual vs Non-virtual:\n";
    Animal& animalRef = dog;
    animalRef.speak();  // Виртуальная: Dog::speak
    animalRef.eat();    // Невиртуальная: Animal::eat
    
    // 5. Коллекция полиморфных объектов (через указатели)
    std::cout << "\nPolymorphic collection:\n";
    std::vector<Animal*> animals;
    animals.push_back(new Dog("Buddy", "Labrador"));
    animals.push_back(new Cat("Felix"));
    animals.push_back(new Cow("Daisy"));
    
    for (const auto* animal : animals) {
        animal->speak();  // Каждый говорит по-своему
    }
    
    // Очистка (виртуальные деструкторы сработают корректно)
    for (auto* animal : animals) {
        delete animal;
    }
    
    // 6. Умные указатели для автоматического управления памятью
    std::cout << "\nSmart pointers:\n";
    std::vector<std::unique_ptr<Animal>> smartAnimals;
    smartAnimals.push_back(std::make_unique<Dog>("Max", "Beagle"));
    smartAnimals.push_back(std::make_unique<Cat>("Luna"));
    
    for (const auto& animal : smartAnimals) {
        animal->speak();
    }
    // Автоматическое удаление
    
    return 0;
}
```

### Ключевые выводы

1. **Полиморфизм** позволяет обращаться с объектами разных типов через единый интерфейс, а правильная реализация определяется во время выполнения

2. **Виртуальные функции** — основа динамического полиморфизма:
   - Объявляются с `virtual` в базовом классе
   - Переопределяются в производных классах
   - Вызов через указатель/ссылку на базовый класс дает поведение производного

3. **Ключевые слова `override` и `final`** помогают избежать ошибок при переопределении

4. **Абстрактные классы** (с чисто виртуальными функциями `= 0`) служат интерфейсами и не могут быть инстанциированы

5. **Виртуальный деструктор** обязателен в полиморфных базовых классах для корректного освобождения ресурсов

6. **RTTI и `dynamic_cast`** позволяют безопасно работать с типами во время выполнения:
   - `typeid` — информация о типе
   - `dynamic_cast` — безопасное приведение (возвращает `nullptr` или кидает `bad_cast`)

7. **Производительность**: виртуальные функции добавляют небольшое накладные расходы (один дополнительный уровень косвенности), но обычно это оправдано гибкостью кода


#### 9.1 Основная идея шаблонов

**Шаблоны** — это механизм C++, позволяющий писать обобщенный код, не зависящий от конкретных типов данных. Компилятор генерирует конкретные версии функций или классов на основе переданных типов **на этапе компиляции**.

**Ключевые понятия:**
- **Параметрический полиморфизм** — возможность использовать один и тот же код с разными типами
- **Инстанцирование шаблона** — процесс генерации компилятором конкретной функции/класса для определенного типа
- **Параметры шаблона** — типы (`typename T`), значения (нетиповые параметры), другие шаблоны

**Синтаксис:**
```cpp
template <typename T>           // Шаблон функции
T max(T a, T b) { return a > b ? a : b; }

template <class T>              // Эквивалентно typename
class Vector {
    T* data;
    // ...
};
```

**Преимущества:**
- Переиспользование кода без потери производительности
- Типобезопасность (ошибки типов обнаруживаются на этапе компиляции)
- Возможность создания контейнеров и алгоритмов, работающих с любыми типами

**Недостатки:**
- Увеличение времени компиляции
- Размер кода (каждый инстанцированный шаблон генерирует отдельный код)
- Сложность чтения сообщений об ошибках

#### 9.2 Перегрузка шаблонных функций

**Правила перегрузки:**
1. Можно перегружать шаблонные функции обычными функциями
2. Можно перегружать шаблонные функции другими шаблонными функциями
3. При вызове компилятор выбирает наиболее подходящий вариант:
   - Обычная функция считается **более точным** соответствием, чем инстанцирование шаблона
   - Специализация шаблона считается точнее общего шаблона
   - Точное соответствие типов лучше, чем приведение

#### 9.3 Специализация шаблонов

**Полная специализация** — определение реализации шаблона для конкретного типа:
```cpp
template <>
class Vector<bool> { /* оптимизированная реализация */ };
```

**Частичная специализация** — определение реализации для семейства типов:
```cpp
template <typename T>
class Vector<T*> { /* реализация для указателей */ };
```

**Когда использовать специализацию:**
- Оптимизация для конкретных типов (например, `vector<bool>`)
- Различное поведение для разных категорий типов (указатели, ссылки)
- Обработка специальных случаев

1. **Типовые параметры**:

2. **Нетиповые параметры** (известны на этапе компиляции):
   - Целочисленные типы (`int`, `size_t`, `char`, `bool`, `enum`)
   - Указатели и ссылки на объекты/функции

3. **Параметры-шаблоны**:

## 9.6 Зависимые имена в C++

``` cpp

template<typename T>
struct S {
    using A = int; // A — это тип (type alias)
};

template<typename T>
void f() {
    S<T>::A* x; // Что это: умножение или объявление указателя?
    // declaration or expression
}

int main() {
    f<int>();
    return 0;
}
```

шаблон - семантика

### 1. Суть проблемы: Умножение или Указатель?

Когда компилятор анализирует шаблонную функцию `f`, он доходит до строки: `S<T>::A* x;`. Возникает фундаментальная синтаксическая неоднозначность. Эту строку можно интерпретировать двумя абсолютно разными способами:

- **Вариант А (Объявление типа):** `S<T>::A` является типом данных. В этом случае мы объявляем локальную переменную `x`, которая является указателем на этот тип.
    
- **Вариант Б (Выражение умножения):** `S<T>::A` является статической переменной (членом класса). В этом случае мы берем эту переменную и умножаем (`*`) её на какую-то другую переменную `x` (которая, возможно, объявлена глобально).
    

Поскольку конкретный тип `T` еще неизвестен (он будет известен только когда мы вызовем `f<int>()`), компилятор не может заглянуть внутрь `S<T>` и проверить, чем именно является `A`.

### 2. Зависимые имена и Правило компилятора

Имя `A` в конструкции `S<T>::A` называется **зависимым именем** (dependent name), так как его смысл напрямую зависит от параметра шаблона `T`.

Чтобы разрешить описанную выше неоднозначность, стандарт C++ устанавливает жесткое правило: **по умолчанию компилятор считает любое зависимое имя значением (переменной/функцией), а не типом.**

Следовательно, строку `S<T>::A* x;` компилятор парсит как математическое выражение умножения. Если переменная `x` до этого не была объявлена в области видимости, вы получите ошибку компиляции вида `error: 'x' was not declared in this scope`.

### 3. Решение: Ключевое слово `typename`

Чтобы сказать компилятору: _"считай это типом, а не значением"_, необходимо использовать ключевое слово `typename`.

Оно ставится непосредственно перед зависимым именем:

``` cpp
template<typename T>
void f() {
    typename S<T>::A* x; // Теперь компилятор точно знает, что это объявление указателя
}
```

_Примечание:_ Использовать `typename` нужно **только** для зависимых имен. Если имя не зависит от параметра шаблона (например, `std::string* s;`), писать `typename` не нужно и даже ошибочно.

## 10.4 Исключения при компилировании

В классическом понимании **исключение (exception)** — это механизм обработки ошибок **времени выполнения** (runtime). Программа не может "выбросить" `std::exception` во время компиляции.

Однако существуют механизмы, которые выполняют аналогичную роль на этапе сборки:

### 1. static_assert (C++11 и выше)

Это «исключение» этапа компиляции. Если условие ложно, компилятор прерывает сборку и выводит сообщение.

C++

```
static_assert(sizeof(void*) == 8, "Поддерживаются только 64-битные системы!");
```

### 2. Ошибки в constexpr и consteval

Если функция помечена как `constexpr`, она может вызываться во время компиляции. Если внутри такой функции логика приводит к генерации исключения (`throw`), компилятор выдаст ошибку сборки, так как механизм `throw/catch` не поддерживается в контексте времени компиляции.

### 3. Ошибки в шаблонах (SFINAE и Concepts)

При работе с шаблонами вместо исключений используется механизм **Concepts** (C++20). Если тип не соответствует требованиям, компилятор выдает ошибку, не пытаясь создать (инстанцировать) неверный код.

---

### Резюме по иерархии catch

Чтобы не допустить логических ошибок, следуйте этой структуре:

1. **Специфичные ошибки** (напр., `catch (const MySubError& e)`)
    
2. **Общие ошибки** (напр., `catch (const std::exception& e)`)
    
3. **Ловушка для всего остального** (напр., `catch (...)`) — используйте с осторожностью.


## 10.5 Исключения в конструкторах и идиома Райля

### Исключения в конструкторах: В чем кроется опасность?

Главное правило C++, которое нужно помнить: **если конструктор выбрасывает исключение, объект считается не до конца созданным, и его деструктор вызван не будет.**

Это порождает серьезную проблему управления ресурсами:

- Если внутри конструктора вы успели выделить память (например, через `new`), открыть файл или захватить мьютекс, а _затем_ произошло исключение, этот ресурс останется в системе навсегда.
    
- Поскольку деструктор для неполного объекта не вызывается, код очистки (например, `delete`) никогда не сработает. Возникает утечка памяти.
    

C++ гарантирует только одно: при возникновении исключения будут вызваны деструкторы для всех **уже полностью сконструированных** подобъектов (полей класса и базовых классов).

---

### Идиома RAII (РАИИ) как элегантное решение

Идиома RAII (Resource Acquisition Is Initialization — Получение ресурса есть инициализация) сводится к простой, но мощной идее: **связать жизненный цикл любого ресурса с жизненным циклом локального объекта.**

1. **Получение ресурса** (выделение памяти, открытие файла) происходит в конструкторе объекта-обертки.
    
2. **Освобождение ресурса** строго инкапсулируется в деструкторе этого же объекта.
    

Поскольку стандарт гарантирует, что при раскрутке стека (stack unwinding) из-за исключения деструкторы всех локальных и _уже полностью созданных_ объектов будут вызваны автоматически, ресурсы всегда корректно освобождаются.

#### Пример проблемы (Утечка памяти):

C++

```
class BadClass {
    int* data1;
    int* data2;
public:
    BadClass() {
        data1 = new int[100];
        // Если здесь выбросится исключение, указатель data1 будет потерян, а память утечет!
        data2 = new int[100]; // Представим, что new бросил std::bad_alloc
    }
    ~BadClass() {
        delete[] data1;
        delete[] data2;
    }
};
```

#### Решение с помощью RAII:

Вместо "голых" указателей (raw pointers) мы используем объекты, которые сами управляют своей памятью — например, умные указатели (`std::unique_ptr`) или стандартные контейнеры (`std::vector`).

C++

```
#include <memory>
#include <vector>

class GoodClass {
    std::unique_ptr<int[]> data1;
    std::vector<int> data2;
public:
    GoodClass() {
        data1 = std::make_unique<int[]>(100);
        // Даже если следующая строка бросит исключение,
        // деструктор полностью созданного поля data1 будет вызван автоматически.
        data2 = std::vector<int>(100);
    }
    // Нам даже не нужно явно писать деструктор ~GoodClass(),
    // поля-обертки очистят за собой память сами.
};
```

### Основные преимущества RAII:

- **Гарантия очистки:** Ресурсы освобождаются при любом выходе из области видимости (штатное завершение функции, `return`, `break`, `continue` или `throw`).
    
- **Чистота кода:** Отпадает необходимость в громоздких и вложенных блоках `try-catch` внутри бизнес-логики класса.
    
- **Инкапсуляция:** Вся логика работы с сырым ресурсом скрыта внутри небольшого класса-менеджера.


## 10.6 Спецификация исключений и ключевое слово noexcept

### Как работает `noexcept`

Ключевое слово `noexcept` сообщает компилятору и программистам, что функция не выпустит исключение за свои пределы.

C++

```
void calculate_eigenvalues() noexcept; // Эта функция гарантирует отсутствие исключений
```

**Что произойдет, если нарушить обещание?** Если внутри функции, помеченной как `noexcept`, все же возникнет исключение и оно попытается покинуть тело функции (т.е. не будет перехвачено внутренним блоком `try-catch`), программа немедленно вызовет `std::terminate()`. Приложение жестко завершится, при этом раскрутка стека и вызов деструкторов **не гарантируются**. Это суровое, но необходимое правило для обеспечения предсказуемости.

---

### Условный `noexcept` и оператор проверки

Спецификация может зависеть от условий этапа компиляции. Это особенно полезно при проектировании обобщенных структур данных (шаблонов).

В C++ `noexcept` выступает в двух ролях:

1. **Спецификатор** (указывает свойство функции).
    
2. **Оператор** (проверяет на этапе компиляции, генерирует ли выражение исключение, возвращая `true` или `false`).
    

C++

```
template <typename T>
void swap_elements(T& a, T& b) noexcept(noexcept(a.swap(b))) {
    a.swap(b);
}
```

В этом примере `swap_elements` будет `noexcept` только в том случае, если метод `swap` у типа `T` сам по себе помечен как `noexcept`.

---

### Практическая польза: Оптимизация и Move-семантика

В контексте сложных вычислений и алгоритмики производительность играет ключевую роль. Использование `noexcept` дает два главных преимущества:

- **Оптимизация компилятора:** Компилятору не нужно генерировать дополнительный код (unwind tables) для отслеживания пути раскрутки стека вокруг вызова такой функции. Это уменьшает размер исполняемого файла и может ускорить работу кода.
    
- **Ускорение стандартных контейнеров (`std::vector`):** Когда динамический массив переполняется, он выделяет новый блок памяти большего размера и переносит туда элементы.
    
    - Если перемещающий конструктор (move constructor) вашего класса помечен как `noexcept`, вектор быстро **переместит** (move) объекты.
        
    - Если `noexcept` отсутствует, вектор из соображений безопасности (чтобы обеспечить строгую гарантию исключений и иметь возможность откатиться при ошибке) будет медленно **копировать** (copy) каждый элемент по одному.

# 11 Итераторы
## 11.1 Основная идея итераторов

Итератор - это то, что ведёт себя как итератор

Основная идея итераторов заключается в **абстрагировании доступа к данным**. Они выступают универсальным «мостом» между контейнерами (структурами данных) и алгоритмами.

Вот почему фраза «ведёт себя как итератор» имеет столь глубокий смысл:

- **Принцип поведения (Duck Typing в шаблонах):** Алгоритму стандартной библиотеки (например, `std::find` или `std::sort`) совершенно не важно, как устроен ваш класс изнутри. Ему важно лишь то, чтобы переданный объект поддерживал определенный интерфейс.
    
- **Сходство с указателями:** Если объект можно разыменовать (`*it`), сдвинуть вперед (`++it`) и сравнить с концом последовательности (`it != end`), компилятор признает его итератором. Поскольку обычные указатели на элементы массива поддерживают все эти операции, они тоже являются полноправными итераторами.
    
- **Разделение логики:** Благодаря этому подходу отпадает необходимость писать отдельную функцию поиска для вектора, отдельную — для списка, и отдельную — для обычного массива. Вы пишете или используете один алгоритм, который работает с _любым_ объектом, который ведёт себя как итератор.

``` cpp
#include <iostream>
#include <vector>
#include <algorithm> // Подключаем для использования алгоритмов, например, std::reverse

int main() {
    // Инициализируем вектор целых чисел
    std::vector<int> numbers = {10, 20, 30, 40, 50};

    std::cout << "Исходный вектор: ";
    
    // 1. Классический цикл с явным указанием типа итератора
    for (std::vector<int>::iterator it = numbers.begin(); it != numbers.end(); ++it) {
        // Разыменовываем итератор, чтобы получить значение элемента
        std::cout << *it << " "; 
    }
    std::cout << "\n";

    // 2. Изменение элементов через итератор с использованием auto (C++11)
    for (auto it = numbers.begin(); it != numbers.end(); ++it) {
        *it += 5; // Увеличиваем каждый элемент на 5
    }

    // 3. Использование итераторов со стандартными алгоритмами
    // std::reverse меняет порядок элементов, принимая итераторы начала и конца
    std::reverse(numbers.begin(), numbers.end());

    std::cout << "Измененный вектор: ";
    
    // 4. Цикл range-based for (под капотом он работает именно через begin() и end()!)
    for (int num : numbers) {
        std::cout << num << " ";
    }
    std::cout << "\n";

    return 0;
}
```

## 11.2 Категории итераторов

### 1. Входные итераторы (Input Iterators)

Умеют только **читать** данные и двигаться только **вперед** (`++it`).

- **Особенность:** Поддерживают только один проход. Прочитав элемент и сдвинувшись вперед, вы не можете гарантированно вернуться назад или прочитать тот же элемент снова.
    
- **Пример:** Итераторы потока ввода (`std::istream_iterator`), читающие данные из файла или консоли.
    

### 2. Выходные итераторы (Output Iterators)

Умеют только **записывать** данные (`*it = value`) и двигаться только **вперед** (`++it`).

- **Особенность:** Тоже одноразовые. Записали значение, сдвинулись дальше.
    
- **Пример:** Итераторы потока вывода (`std::ostream_iterator`) или специальные вставщики (`std::back_inserter`).
    

### 3. Однонаправленные итераторы (Forward Iterators)

Объединяют свойства входных и выходных итераторов. Умеют читать, писать и двигаться **вперед**.

- **Особенность:** Главное отличие от входных/выходных — они поддерживают **многократный проход**. Вы можете сохранить копию итератора и позже пройтись по тем же элементам еще раз.
    
- **Пример:** Итераторы односвязного списка (`std::forward_list`).
    

### 4. Двунаправленные итераторы (Bidirectional Iterators)

Имеют все возможности однонаправленных, но умеют также делать шаг **назад** (`--it`).

- **Особенность:** Позволяют алгоритмам двигаться по контейнеру в обоих направлениях (например, алгоритм `std::reverse`).
    
- **Пример:** Итераторы двусвязного списка (`std::list`), а также деревьев: множеств (`std::set`) и словарей (`std::map`).
    

### 5. Итераторы произвольного доступа (Random Access Iterators)

Самые мощные итераторы. Умеют всё вышеперечисленное, а также **прыгать** на любое расстояние.

- **Особенность:** Поддерживают адресную арифметику (`it + n`, `it - n`), сравнение позиций (`it1 < it2`) и оператор индексации (`it[n]`). По скорости работы и интерфейсу они эквивалентны обычным сырым указателям.
    
- **Пример:** Итераторы вектора (`std::vector`), дека (`std::deque`) и обычных массивов.
    

---

> **Примечание:** В современных стандартах (начиная с C++17 и особенно C++20) была добавлена еще одна категория — **Непрерывные итераторы (Contiguous Iterators)**. Это подвид итераторов произвольного доступа, которые дают строгую гарантию, что элементы в памяти лежат физически вплотную друг к другу (как в `std::vector` или `std::array`).

## 11.3 Функции advance и distance

### Функция `std::advance`

Функция `std::advance(it, n)` сдвигает переданный итератор `it` на `n` позиций. Значение `n` может быть отрицательным, если итератор является двунаправленным или итератором произвольного доступа.

Вся прелесть этой функции заключается в её оптимизации под капотом (с помощью механизмов определения типов во время компиляции):

- Если передать ей **итератор произвольного доступа** (например, от `std::vector`), она просто выполнит операцию `it += n`. Вычислительная сложность составит $O(1)$.
    
- Если передать ей **другой итератор** (например, от `std::list`), она запустит цикл, который сделает `++it` ровно `n` раз. Сложность составит $O(N)$.
    

### Функция `std::distance`

Функция `std::distance(first, last)` вычисляет расстояние между двумя итераторами, то есть количество шагов, которое нужно сделать от `first`, чтобы добраться до `last`.

Здесь работает та же логика оптимизации:

- Для **итераторов произвольного доступа** она просто вычтет один указатель из другого: `last - first`. Работает за $O(1)$.
    
- Для **остальных итераторов** она будет в цикле делать инкремент первого итератора, пока тот не станет равен второму, подсчитывая количество шагов. Работает за $O(N)$.
    

---

### Пример использования

Ниже приведен код, демонстрирующий, как один и тот же синтаксис применяется к совершенно разным структурам данных:

C++

```
#include <iostream>
#include <vector>
#include <list>
#include <iterator>

int main() {
    // Работа с вектором (итератор произвольного доступа)
    std::vector<int> vec = {10, 20, 30, 40, 50};
    auto vec_it = vec.begin();
    
    // Прыгаем на 3 позиции вперед за O(1)
    std::advance(vec_it, 3); 
    std::cout << "Элемент вектора: " << *vec_it << "\n"; // Выведет 40
    
    // Считаем расстояние за O(1)
    int vec_dist = std::distance(vec.begin(), vec_it);
    std::cout << "Расстояние в векторе: " << vec_dist << "\n\n"; // Выведет 3

    // Работа со списком (двунаправленный итератор)
    std::list<int> lst = {10, 20, 30, 40, 50};
    auto lst_it = lst.begin();
    
    // Шагаем на 3 позиции вперед в цикле за O(N)
    std::advance(lst_it, 3);
    std::cout << "Элемент списка: " << *lst_it << "\n"; // Выведет 40
    
    // Считаем расстояние в цикле за O(N)
    int lst_dist = std::distance(lst.begin(), lst_it);
    std::cout << "Расстояние в списке: " << lst_dist << "\n"; // Выведет 3

    return 0;
}
```

> **Важное замечание:** Функция `std::advance` изменяет сам переданный ей итератор (поэтому он передается по ссылке). Если вам нужно получить _новый_ итератор, сдвинутый на `n` позиций, не меняя исходный, в современных стандартах C++ (начиная с C++11) используют функции `std::next` и `std::prev`.

## 11.4 Const итераторы

**Константные итераторы (`const_iterator`)** в C++ — это итераторы, которые позволяют обходить контейнер и читать его элементы, но строго запрещают их изменять.

Их поведение полностью аналогично указателю на константное значение (`const T*`). Вы можете перемещать сам итератор (например, делать `++it` или `--it`), но значение, скрытое по этому адресу (`*it`), будет защищено от записи.

### Зачем они нужны?

1. **Безопасность (Const-correctness):** Они гарантируют, что код, задуманный только для чтения данных, случайно их не перезапишет. Это особенно полезно при проектировании надежных интерфейсов.
    
2. **Работа с константными объектами:** Если вы передаете контейнер в функцию по константной ссылке (например, `void process(const std::vector<int>& v)`), вызов `v.begin()` вернет именно `const_iterator`. Получить обычный итератор от константного контейнера нельзя.
    

### Как их получить?

Современный стандарт C++ предоставляет специальные методы `cbegin()`, `cend()`, `crbegin()` и `crend()` для явного получения константных итераторов, даже если сам контейнер не является константным.

### Пример использования

C++

```
#include <iostream>
#include <vector>

void printVector(const std::vector<int>& vec) {
    // Поскольку vec передан по константной ссылке, vec.begin() возвращает const_iterator.
    // Использование auto здесь автоматически выведет тип std::vector<int>::const_iterator
    for (auto it = vec.begin(); it != vec.end(); ++it) {
        std::cout << *it << " ";
        // *it = 10; // Ошибка компиляции: попытка изменить константный элемент
    }
    std::cout << "\n";
}

int main() {
    std::vector<int> numbers = {10, 20, 30, 40, 50};

    // Явное использование cbegin() и cend() для гарантии защиты данных
    for (std::vector<int>::const_iterator it = numbers.cbegin(); it != numbers.cend(); ++it) {
        std::cout << *it << " ";
    }

    return 0;
}
```

### Важное различие: `const_iterator` против `const iterator`

Это классическая ловушка при работе с памятью и указателями в C++:

- **`std::vector<int>::const_iterator it;`**
    
    - **Можно** сдвигать итератор (`++it`).
        
    - **Нельзя** менять значение (`*it = 5`).
        
    - _Аналог в С:_ `const int* p;` (указатель на константу).
        
- **`const std::vector<int>::iterator it = numbers.begin();`**
    
    - **Нельзя** сдвигать итератор (`++it` вызовет ошибку компиляции).
        
    - **Можно** менять значение (`*it = 5`).
        
    - _Аналог в С:_ `int* const p;` (константный указатель).
        

## 11.5 Reverse итератор

**Обратные итераторы (`reverse_iterator`)** — это специальный вид итераторов, которые позволяют обходить контейнер в обратном направлении, от конца к началу. Главная их особенность в том, что операция инкремента (`++`) перемещает их к _предыдущему_ элементу контейнера, а декремента (`--`) — к _следующему_.

В стандартной библиотеке C++ `std::reverse_iterator` — это не написанный с нуля класс, а **адаптер**. Он оборачивает обычный двунаправленный итератор (bidirectional iterator) или итератор произвольного доступа (random access iterator) и инвертирует его логику.

### Главный секрет: сдвиг на единицу (Off-by-one)

Это самая важная концепция для понимания того, как они работают под капотом.

Чтобы алгоритмы STL (например, `std::sort` или `std::find`) могли работать с обратными итераторами точно так же, как с прямыми, необходимо было сохранить концепцию полуоткрытого интервала **`[begin, end)`**.

Поэтому обратный итератор физически хранит указатель на элемент, следующий _после_ того, к которому вы фактически обращаетесь:

- `rbegin()` конструируется из `end()` (то есть указывает за пределы контейнера).
    
- `rend()` конструируется из `begin()` (указывает на первый элемент).
    

Когда вы разыменовываете обратный итератор (`*rit`), внутри него происходит следующее: он берет базовый итератор, создает его временную копию, сдвигает ее на один шаг назад (`--tmp`) и возвращает значение по этому адресу.

**Наглядная схема:**

Plaintext

```
Элементы:          [ 10 ]      [ 20 ]      [ 30 ]      [ 40 ]
Прямые:           ^begin()                            ^end()
Обратные (логика):            ^rend()                 ^rbegin() 
Обратные (база):  ^rend().base()                      ^rbegin().base()
```

### Пример использования

C++

```
#include <iostream>
#include <vector>

int main() {
    std::vector<int> data = {10, 20, 30, 40, 50};

    // Стандартный проход с конца в начало
    std::cout << "Reverse: ";
    for (auto rit = data.rbegin(); rit != data.rend(); ++rit) {
        std::cout << *rit << " "; // Выведет 50 40 30 20 10
    }
    std::cout << "\n";

    return 0;
}
```

### Метод `.base()` и конвертация

Поскольку `reverse_iterator` — это адаптер, вы всегда можете "достать" из него оригинальный прямой итератор с помощью метода `.base()`.

Но здесь нужно быть очень осторожным из-за описанного выше сдвига. Если ваш обратный итератор `rit` логически указывает на значение `30`, то вызов `rit.base()` вернет прямой итератор, указывающий на `40` (на один элемент правее).

Это часто используется при удалении элементов. Метод `erase()` принимает только обычные итераторы. Если вы нашли элемент с помощью обратного итератора и хотите его удалить, нужно сделать поправку:

C++

```
auto rit = std::find(data.rbegin(), data.rend(), 30);
if (rit != data.rend()) {
    // Чтобы удалить 30, нам нужен прямой итератор, указывающий на 30.
    // rit.base() указывает на 40. Поэтому мы сначала "сдвигаем" rit вперед 
    // (что означает логический сдвиг влево, к 20), а его база укажет как раз на 30.
    data.erase((++rit).base()); 
}
```

### Как добавить их в свой класс?

Если вы, мой господин, пишете собственный контейнер (как ваш класс `PUBLISH` или `BigInteger`), вам не нужно писать логику обратного итератора с нуля. Достаточно реализовать обычный `iterator`, а затем использовать стандартный адаптер:

C++

```
using reverse_iterator = std::reverse_iterator<iterator>;
using const_reverse_iterator = std::reverse_iterator<const_iterator>;

reverse_iterator rbegin() { return reverse_iterator(end()); }
reverse_iterator rend() { return reverse_iterator(begin()); }
```

## 11.6 Output итератор

**Итераторы вывода (Output Iterators)** — это итераторы, которые предназначены **исключительно для записи** данных в последовательность. Если `const_iterator` запрещал нам менять элементы, то итератор вывода, наоборот, запрещает их читать.

Их логика работы напоминает запись данных в поток или трубу: вы можете отправить туда значение и продвинуться дальше, но не можете заглянуть внутрь, чтобы узнать, что там уже лежит, или вернуться назад.

### Главные особенности и ограничения

1. **Только запись (Write-only):** Операция разыменования допускается только в левой части оператора присваивания.
    
    - `*it = 10;` — **Можно.**
        
    - `int x = *it;` — **Нельзя** (поведение не определено).
        
2. **Однопроходность (Single-pass):** Вы можете пройти по последовательности только один раз и двигаться только вперед (`++it` или `it++`).
    
3. **Строгое чередование:** В правильном коде за каждой операцией записи `*it = value` должно следовать продвижение итератора `++it`. Записывать дважды в одну и ту же позицию без сдвига или сдвигать без записи — плохая практика (хотя некоторые реализации это прощают, стандарт этого не гарантирует).
    

### Где они используются?

Чаще всего итераторы вывода используются в качестве **приемника данных** (destination) в стандартных алгоритмах STL, таких как `std::copy`, `std::transform`, `std::generate`.

Стандартная библиотека C++ предоставляет несколько невероятно удобных готовых итераторов вывода:

#### 1. Инсертеры (Insert Iterators)

Обычный итератор при записи перезаписывает существующий элемент (например, память должна быть уже выделена). Инсертеры же под капотом вызывают методы добавления элементов (`push_back`, `push_front` или `insert`), автоматически расширяя контейнер.

- `std::back_inserter(container)` — при записи делает `container.push_back(value)`.
    
- `std::front_inserter(container)` — делает `container.push_front(value)`.
    
- `std::inserter(container, it)` — делает `container.insert(it, value)`.
    

#### 2. Потоковые итераторы (Stream Iterators)

Позволяют алгоритмам STL писать данные напрямую в потоки ввода-вывода (в консоль или в файл), как если бы это был обычный контейнер.

- `std::ostream_iterator<T>(stream, delimiter)`
    

### Наглядный пример

C++

```
#include <iostream>
#include <vector>
#include <algorithm>
#include <iterator>

int main() {
    std::vector<int> source = {10, 20, 30, 40, 50};
    std::vector<int> destination;

    // ОШИБКА: std::copy(source.begin(), source.end(), destination.begin());
    // destination пуст, памяти нет. Обычный итератор попытается перезаписать несуществующую память.

    // ПРАВИЛЬНО: используем std::back_inserter (Output Iterator)
    // На каждой итерации он будет брать элемент из source и делать push_back в destination
    std::copy(source.begin(), source.end(), std::back_inserter(destination));

    // Используем std::ostream_iterator (еще один Output Iterator) для вывода на экран.
    // Он будет писать каждый полученный элемент в std::cout, добавляя " | " после каждого.
    std::ostream_iterator<int> out_it(std::cout, " | ");
    
    // Копируем данные из destination напрямую в консоль!
    std::copy(destination.begin(), destination.end(), out_it);
    // Вывод: 10 | 20 | 30 | 40 | 50 | 

    return 0;
}
```


## 11.7 Stream итератор

**Потоковые итераторы (Stream Iterators)** — это изящный механизм стандартной библиотеки C++, который стирает границу между контейнерами данных (как `std::vector` или `std::list`) и потоками ввода-вывода (как консоль `std::cin`/`std::cout` или файлы `std::ifstream`/`std::ofstream`).

Благодаря им вы можете применять мощные алгоритмы STL (например, `std::copy`, `std::count`, `std::transform`) напрямую к потокам, вообще не написав ни одного ручного цикла. Итераторы сами "вытягивают" или "проталкивают" данные через поток.

Существует два основных класса потоковых итераторов, определенных в заголовочном файле `<iterator>`:

### 1. `std::istream_iterator` (Итератор ввода)

Относится к категории **Input Iterators**. Он позволяет читать данные из входного потока (`std::istream`) последовательно, элемент за элементом.

- **Как он читает:** Под капотом он использует стандартный перегруженный оператор `>>`. Это означает, что он автоматически пропускает пробелы, табуляции и переносы строк, читая форматированные данные (числа, слова и т.д.).
    
- **Как определить конец (End-of-File):** Любому алгоритму STL нужны начало и конец диапазона `[begin, end)`. Но у потока нет метода `end()`. Для этого используется специальный **итератор конца потока (end-of-stream iterator)**, который создается вызовом конструктора по умолчанию: `std::istream_iterator<T>()`. Когда поток достигает конца файла (EOF) или сталкивается с ошибкой чтения (например, ожидалось число, а введена буква), рабочий итератор становится равным итератору конца потока.
    

### 2. `std::ostream_iterator` (Итератор вывода)

Относится к категории **Output Iterators** (мы кратко затрагивали его ранее). Предназначен для записи данных в выходной поток (`std::ostream`).

- **Как он пишет:** Использует оператор `<<`.
    
- **Форматирование:** В конструктор можно передать строку-разделитель, которая будет автоматически добавляться в поток после записи каждого элемента.
    

---

### Наглядный пример: Элегантная работа с вводом-выводом

В этом примере мы считаем неизвестное количество чисел из консоли, отсортируем их и выведем обратно, используя минимум кода.

C++

```
#include <iostream>
#include <vector>
#include <iterator>
#include <algorithm>

int main() {
    std::cout << "Введите числа (для завершения введите любую букву или нажмите Ctrl+D/Ctrl+Z):\n";

    // 1. Создаем итераторы ввода, привязанные к потоку cin
    std::istream_iterator<int> cin_it(std::cin);
    std::istream_iterator<int> eos; // Итератор конца потока (End-Of-Stream)

    // 2. Инициализируем вектор диапазоном итераторов.
    // Вектор будет "вытягивать" числа из cin, пока не встретит EOF или ошибку ввода.
    std::vector<int> numbers(cin_it, eos);

    // 3. Обрабатываем данные (например, сортируем)
    std::sort(numbers.begin(), numbers.end());

    // 4. Выводим результат в консоль через потоковый итератор вывода
    std::cout << "Отсортированные числа:\n";
    std::ostream_iterator<int> cout_it(std::cout, " -> ");
    
    // Копируем данные из вектора прямо в консоль
    std::copy(numbers.begin(), numbers.end(), cout_it);
    std::cout << "Конец.\n";

    return 0;
}
```

### Архитектурные тонкости (Под капотом)

При работе с потоковыми итераторами в C++ есть несколько важных технических нюансов:

- **Ленивые вычисления (Lazy Evaluation) и предчтение:** Когда вы создаете `std::istream_iterator<int>(std::cin)`, итератор _сразу же_ пытается считать первое значение из потока и кэширует его внутри себя. Это необходимо для того, чтобы сразу проверить, не был ли поток пустым изначально (чтобы корректно сравнить его с `end()`). Операция `++it` просто считывает следующее значение в кэш, а `*it` возвращает текущее закэшированное значение.
    
- **Проблема "Most Vexing Parse":** Если вы попытаетесь инициализировать вектор анонимными итераторами вот так:
    
    `std::vector<int> v(std::istream_iterator<int>(std::cin), std::istream_iterator<int>());`
    
    Компилятор C++ может интерпретировать эту строку не как создание переменной `v`, а как _объявление функции_ `v`, которая возвращает `std::vector<int>` и принимает два хитрых аргумента. Чтобы этого избежать, используют фигурные скобки:
    
    `std::vector<int> v{std::istream_iterator<int>(std::cin), std::istream_iterator<int>()};`
    
- **Буферизация файлов:** Для быстрой работы с бинарными файлами `std::istream_iterator` не подходит (так как он использует форматированный ввод через `>>`). Для побайтового чтения файлов в STL существуют `std::istreambuf_iterator` и `std::ostreambuf_iterator`, которые работают напрямую с буфером потока, минуя тяжелую логику форматирования.

# Вектор bool

Чтобы упаковать 8 логических значений в один байт, мы не можем вернуть `bool&` (ссылку на бит). Мы должны вернуть специальный объект, который знает, где лежит нужный байт и какой именно бит в нем нужно изменить.

``` cpp
#include <iostream>
#include <vector>
#include <cstdint> // Для uint8_t (гарантированно 1 байт)

class MyVectorBool {
private:
    // Под капотом мы храним сырые байты. 
    // Один uint8_t вмещает 8 значений bool.
    std::vector<uint8_t> data;
    size_t bit_size;

public:
    class BitReference {
    private:
        uint8_t& byte; // Ссылка на реальный байт в памяти вектора
        uint8_t mask;  // Маска нужного бита (например, 00001000)

    public:
        // Конструктор доступен классу MyVectorBool
        BitReference(uint8_t& target_byte, uint8_t bit_position) 
            : byte(target_byte), mask(1 << bit_position) {}

        // 1. ОПЕРАЦИЯ ЧТЕНИЯ: Неявное приведение прокси-объекта к типу bool
        // Вызывается, когда мы делаем: bool b = my_vec[0];
        operator bool() const {
            return (byte & mask) != 0;
        }

        // 2. ОПЕРАЦИЯ ЗАПИСИ: Перегрузка оператора присваивания
        // Вызывается, когда мы делаем: my_vec[0] = true;
        BitReference& operator=(bool value) {
            if (value) {
                byte |= mask;  // Устанавливаем бит в 1 (побитовое ИЛИ)
            } else {
                byte &= ~mask; // Сбрасываем бит в 0 (побитовое И с инверсией)
            }
            return *this;
        }

        // 3. ПРИСВАИВАНИЕ ДРУГ ДРУГУ: my_vec[0] = my_vec[1];
        BitReference& operator=(const BitReference& other) {
            return *this = static_cast<bool>(other); 
        }
    };
    // =========================================================

    // Конструктор вектора
    MyVectorBool(size_t size) : bit_size(size) {
        // Выделяем память: делим количество бит на 8 (округляя вверх)
        size_t byte_count = (size + 7) / 8;
        data.resize(byte_count, 0); // Заполняем нулями
    }

    // Доступ для изменения (возвращает прокси-объект по значению!)
    BitReference operator[](size_t index) {
        size_t byte_index = index / 8;  // В каком байте лежит бит
        uint8_t bit_index = index % 8;  // Какой это бит по счету (от 0 до 7)
        
        return BitReference(data[byte_index], bit_index);
    }

    // Доступ только для чтения (константный метод)
    // Здесь прокси не нужен, мы просто возвращаем скопированный bool
    bool operator[](size_t index) const {
        size_t byte_index = index / 8;
        uint8_t bit_index = index % 8;
        return (data[byte_index] & (1 << bit_index)) != 0;
    }
};

// --- Пример использования ---
int main() {
    MyVectorBool flags(10); // Выделится всего 2 байта памяти (16 бит)

    flags[2] = true;  // Вызов BitReference::operator=(bool)
    flags[5] = true;

    // Вызов BitReference::operator bool()
    if (flags[2]) {
        std::cout << "Бит 2 установлен!\n";
    }

    std::cout << "Флаги: ";
    for (size_t i = 0; i < 10; ++i) {
        std::cout << flags[i] << " "; 
    }
    std::cout << "\n";

    return 0;
}
```

### Разбор магии (Как это работает)

Вся суть заключена в методе `operator[]`. Обратите внимание: он возвращает `BitReference`, а **не** `BitReference&` (не ссылку).

Когда вы пишете в коде:

`flags[2] = true;`

Для компилятора это разбивается на два шага:

1. Вызов `flags.operator[](2)`. Этот метод вычисляет, что 2-й бит лежит в 0-м байте. Создается временный объект `BitReference`, который хранит ссылку на 0-й байт и маску `00000100` (число 4). Объект возвращается.
    
2. У этого временно созданного объекта `BitReference` вызывается `operator=(true)`. Объект берет свой байт, накладывает маску и изменяет нужный бит в памяти вектора. После этой строки кода временный прокси-объект благополучно уничтожается.
    

Когда вы пишете:

`bool state = flags[2];`

Происходит следующее:

1. Вектор возвращает тот же самый временный прокси-объект.
    
2. Прокси-объект пытается присвоиться переменной типа `bool`. Компилятор видит это и ищет способ преобразования. Он находит перегруженный `operator bool() const` в классе `BitReference`, вызывает его, читает бит из памяти и возвращает настоящий `bool`.
    

### Почему это ломает итераторы (Еще раз, с высоты кода)

Теперь, когда вы видите код `BitReference`, представьте, что вы пишете для этого класса итератор.

Обычный итератор при вызове `*it` должен возвращать ссылку `T&` (в нашем случае `bool&`).

Но мы не можем вернуть `bool&` из нашего итератора, потому что физического `bool` в памяти нет! Мы вынуждены возвращать `BitReference`. А возвращение прокси-объекта по значению вместо истинной ссылки нарушает контракт стандартных итераторов С++.

# Deque

`std::deque` (читается как "дек" — от double-ended queue, двусторонняя очередь).

Это, пожалуй, самый недооцененный и архитектурно интересный последовательный контейнер в C++. Многие студенты ошибочно думают, что это либо разновидность двусвязного списка (`std::list`), либо просто `std::vector`, который умеет расти в обе стороны. И то, и другое — неверно.

`std::deque` — это гибрид, который пытается взять лучшее от непрерывных массивов и списков.

### Как `deque` устроен под капотом?

В отличие от вектора, который хранит все свои элементы в одном гигантском непрерывном куске памяти, **дек хранит элементы в виде набора фиксированных блоков (чанков)**.

Архитектура дека состоит из двух уровней:

1. **Мап (Map/Directory):** Небольшой непрерывный массив указателей.
    
2. **Блоки (Chunks/Buffers):** Одинаковые по размеру массивы в куче, где реально лежат ваши данные.
    

**Визуализация структуры:**


``` Plaintext
               Центральный Массив Указателей (Map)
               [ * ]   [ * ]   [ * ]   [ * ]   [   ]  (резерв для роста)
                 |       |       |       |
      +----------+       |       |       +----------+
      |                  |       |                  |
      V                  V       V                  V
Блок 1 (Начало)        Блок 2  Блок 3         Блок 4 (Конец)
[ . | . | 10 | 20 ]   [ 30 | 40 | 50 | 60 ]  [ 70 | 80 | . | . ]
          ^                                            ^
        begin()                                      end()
```

_(Точки `.` обозначают свободную память внутри крайних блоков)._

### Как он растет? (Почему `push_front` работает за $O(1)$)

Если вы делаете `push_back`, а последний блок заполнен, дек не копирует старые данные (как это делает вектор при переполнении `capacity`). Он просто:

1. Выделяет в памяти новый пустой блок.
    
2. Записывает указатель на этот блок в следующую свободную ячейку Мапа.
    

Если вы делаете `push_front`, происходит то же самое, но в начале: добавляется новый блок, и указатель пишется в начало Мапа.

Если сам Мап переполняется указателями, перевыделяется только Мап. Сами блоки с данными остаются на своих местах, **ни один элемент не копируется и не перемещается**.

### Магия `operator[]` за $O(1)$

Поскольку размер каждого блока (пусть это будет $K$) строго фиксирован, дек может вычислить адрес любого элемента за константное время $O(1)$, используя простую математику, которая так близка вашему профилю.

Чтобы найти элемент с индексом $i$:

1. Индекс нужного блока в Мапе: `map_index = i / K`
    
2. Смещение внутри этого блока: `offset = i % K`
    

Это работает за $O(1)$, но требует двух обращений к памяти (сначала получить указатель на блок из Мапа, потом элемент из блока), плюс операции деления и взятия остатка. Поэтому `deque[i]` всегда **немного медленнее**, чем `vector[i]`, где происходит лишь одно сложение указателя.

### Главная суперсила: Инвалидация ссылок

Это вопрос, на котором "валят" на собеседованиях уровня Middle/Senior C++.

У `std::vector` есть проблема: если вы взяли указатель или ссылку на элемент, а затем сделали `push_back` (и произошла реаллокация), ваш указатель становится **висячим** (dangling).

У `std::deque` правило другое: **Добавление элементов в начало или конец дека НИКОГДА не инвалидирует указатели и ссылки на существующие элементы.**

_(Примечание: сами итераторы при этом могут инвалидироваться, так как может перевыделиться центральный Мап, но данные, на которые вы ссылались, никуда из своих блоков не уедут)._

### Вектор против Дека: Что выбрать?

|**Характеристика**|**std::vector**|**std::std::deque**|
|---|---|---|
|**Добавление в конец (`push_back`)**|$O(1)$ амортизированное (иногда тяжелые реаллокации)|$O(1)$ строгое (всегда быстро, выделяется малый блок)|
|**Добавление в начало (`push_front`)**|$O(N)$ (сдвиг всех элементов)|$O(1)$|
|**Доступ по индексу (`[i]`)**|Самый быстрый|Очень быстрый, но медленнее вектора|
|**Кэш-френдли (Locality of reference)**|Идеально (всё рядом)|Хорошо, но прыгает между блоками|

Именно из-за отсутствия тяжелых реаллокаций и строгой гарантии $O(1)$ для вставки по краям, `std::deque` является **контейнером по умолчанию** под капотом у адаптеров `std::stack` и `std::queue`.

``` cpp
#include <iostream>
#include <map>
#include <string>

int main() {
    // Карта: имя студента (ключ) -> оценка (значение)
    std::map<std::string, int> grades;

    grades["Иван"] = 4;
    grades["Анна"] = 5;
    grades["Борис"] = 3;

    // Итератор по умолчанию пройдет по ключам в алфавитном порядке: Анна, Борис, Иван
    for (std::map<std::string, int>::iterator it = grades.begin(); it != grades.end(); ++it) {
```

# Stack, Queue, Priority_queue

В стандартной библиотеке C++ `std::stack`, `std::queue` и `std::priority_queue` — это **не самостоятельные контейнеры**, а **адаптеры контейнеров (Container Adapters)**.

Они не управляют памятью сами. Вместо этого они берут существующий базовый контейнер (тот же вектор или дек) и "обрезают" его интерфейс, заставляя работать по строгим правилам конкретной структуры данных.

Именно поэтому **ни у одного из этих трех адаптеров нет итераторов**. Концепция итератора подразумевает свободный обход элементов, что прямо нарушает саму суть стека или очереди.

Давайте разберем их архитектуру под капотом.

---

### 1. `std::stack` (Стек)

- **Принцип:** LIFO (Last In, First Out — Последним пришел, первым ушел).
    
- **Контейнер под капотом по умолчанию:** `std::deque`.
    
- **Основные методы:** `push()`, `pop()`, `top()`.
    

**Почему по умолчанию используется `deque`, а не `vector`?**

Стек можно реализовать поверх вектора (`std::stack<int, std::vector<int>>`). Но дек выбран по умолчанию, потому что при добавлении большого количества элементов вектор периодически делает реаллокацию и копирует все старые элементы в новую память. Дек же просто выделяет новый блок памяти, ничего не копируя. Для стека, который может стремительно расти, дек обеспечивает более стабильную производительность $O(1)$ при `push()`.

### 2. `std::queue` (Очередь)

- **Принцип:** FIFO (First In, First Out — Первым пришел, первым ушел).
    
- **Контейнер под капотом по умолчанию:** `std::deque`.
    
- **Основные методы:** `push()`, `pop()`, `front()`, `back()`.
    

**Почему нельзя использовать `vector`?**

В отличие от стека, вы **не можете** создать очередь поверх `std::vector`. Очереди требуется быстро удалять элементы из начала (`pop_front`). У вектора удаление первого элемента занимает $O(N)$, так как нужно сдвинуть весь оставшийся массив влево. У дека удаление из начала работает за строгое $O(1)$.

### 3. `std::priority_queue` (Очередь с приоритетом)

Это самый интересный и сложный из адаптеров. Он не подчиняется ни LIFO, ни FIFO.

- **Принцип:** Элемент с наивысшим приоритетом извлекается первым.
    
- **Контейнер под капотом по умолчанию:** `std::vector`.
    
- **Алгоритм под капотом:** Бинарная куча (Binary Heap).
    

**Как это работает в памяти?**

Чтобы быстро находить максимальный элемент, `priority_queue` не сортирует весь вектор целиком (это было бы слишком долго — $O(N \log N)$ на каждую вставку). Вместо этого она использует алгоритмы `<algorithm>`: `std::make_heap`, `std::push_heap` и `std::pop_heap`.

Они выстраивают элементы в векторе так, чтобы они формировали математическую структуру **двоичной кучи**.

- **Вставка (`push`)** добавляет элемент в конец вектора и делает "просеивание вверх" (sift-up). Время: $O(\log N)$.
    
- **Извлечение (`pop`)** меняет первый и последний элементы местами, удаляет последний и делает "просеивание вниз" (sift-down) для нового первого элемента. Время: $O(\log N)$.
    
- **Просмотр максимума (`top`)** — это просто обращение к нулевому элементу вектора `vec[0]`. Время: $O(1)$.
    

По умолчанию `priority_queue` использует компаратор `std::less<T>`, что делает ее **Max-Heap** (кучей максимумов) — на вершине всегда будет самое большое число.

---

Я подготовил для вас визуальную панель, чтобы вы могли своими глазами увидеть, как одно и то же действие по-разному распределяет элементы в этих трех структурах.

### Как использовать `priority_queue` для минимумов?

Поскольку по умолчанию это Max-Heap, многих ставит в тупик вопрос: как сделать так, чтобы минимальный элемент был первым (Min-Heap)? Для этого нужно передать правильный компаратор. Шаблон требует указать все три параметра: тип данных, тип базового контейнера и тип компаратора:


``` cpp
#include <iostream>
#include <queue>
#include <vector>
#include <functional> // Для std::greater

int main() {
    // std::greater переворачивает логику: на вершине будет минимальный элемент
    std::priority_queue<int, std::vector<int>, std::greater<int>> min_heap;

    min_heap.push(50);
    min_heap.push(10);
    min_heap.push(30);

    std::cout << min_heap.top(); // Выведет 10
    
    return 0;
}
```

``` cpp
#include <iostream>

// Наша пользовательская структура данных

    int priority;
    std::string name;

    // 1. Создаем лямбда-выражение для сравнения.
    // Если мы хотим Min-Heap (наименьшие значения наверху), 
    // компаратор должен возвращать true, если 'a' ХУЖЕ (больше) 'b'.
    auto cmp = [](const Task& a, const Task& b) {
        return a.priority > b.priority; 
    };

    // 2. Объявляем очередь.
    // Внимание на decltype(cmp): шаблон требует ТИП компаратора, а лямбда - это ОБЪЕКТ.
    // Поэтому мы извлекаем тип лямбды, а саму лямбду передаем в конструктор очереди.
    std::priority_queue<Task, std::vector<Task>, decltype(cmp)> sys_queue(cmp);

    // Загружаем задачи в случайном порядке
    sys_queue.push({5, "Фоновая дефрагментация"});
    sys_queue.push({0, "Обработка прерывания (КРИТИЧНО)"});
    sys_queue.push({2, "Отрисовка пользовательского интерфейса"});

    std::cout << "--- Выполнение задач ---\n";
    
    // Извлекаем задачи. Они автоматически выстроятся по приоритету!
    while (!sys_queue.empty()) {
        const Task& current = sys_queue.top();
        std::cout << "[Приоритет " << current.priority << "] Выполняется: " << current.name << "\n";
        sys_queue.pop();
    }

    return 0;
}
```

``` cpp
    Range(int start, int end) : start_val(start), end_val(end) {}

    // Наш собственный итератор (Категория: Forward Iterator)
    // ========================================================
    class Iterator {
    private:
        int current;

    public:
        // 1. Обязательные алиасы типов (Iterator traits)
        // Без них стандартные алгоритмы вроде std::find откажутся работать с нашим итератором
        using iterator_category = std::forward_iterator_tag; // Идем только вперед
        using value_type        = int;
        using difference_type   = std::ptrdiff_t;
        using pointer           = int*;
        using reference         = int&;

        // Конструктор итератора
        explicit Iterator(int val) : current(val) {}

        // 2. Разыменование: возвращаем текущее значение
        int operator*() const { 
            return current; 
        }

        // 3. Префиксный инкремент (++it) - двигаемся вперед
        Iterator& operator++() {
            ++current;
            return *this;
        }

        // Постфиксный инкремент (it++)
        Iterator operator++(int) {
            Iterator tmp = *this;
            ++(*this);
            return tmp;
        }

        // 4. Операторы сравнения (необходимы для проверки it != end())
        friend bool operator==(const Iterator& a, const Iterator& b) { 
            return a.current == b.current; 
        }
        friend bool operator!=(const Iterator& a, const Iterator& b) { 
            return a.current != b.current; 
        }
    };
```

# List

Следующая классическая структура данных в STL — это `std::list` (двусвязный список).

В отличие от `vector` и `deque`, которые пытаются хранить данные кучно, `std::list` — это **узловой контейнер (node-based container)**.

### Архитектура под капотом

Каждый элемент в `std::list` — это отдельный объект в куче (heap), который называется узлом (node). Узел хранит в себе три вещи:

1. Само значение (полезная нагрузка).
    
2. Указатель на следующий узел (`next`).
    
3. Указатель на предыдущий узел (`prev`).
    

Это дает ему фундаментальные теоретические отличия от массивов:

- **Идеальная вставка и удаление:** Если у вас уже есть итератор, указывающий на нужный элемент, удаление (`erase`) или вставка перед ним (`insert`) занимает строгое время **$O(1)$**. Никаких сдвигов элементов, как в векторе, не происходит — просто переписываются пара указателей.
    
- **Стабильность итераторов:** При добавлении или удалении элементов в списке **ни один итератор не инвалидируется** (кроме итератора на тот элемент, который вы конкретно удаляете).
    

### Итераторы списка

Итератор `std::list` относится к категории **Двунаправленных итераторов (Bidirectional Iterators)**.

Поскольку узлы разбросаны по памяти случайным образом, физически невозможно вычислить адрес пятого элемента, зная адрес первого.

- `++it` — работает (переход по указателю `next`).
    
- `--it` — работает (переход по указателю `prev`).
    
- `it += 5` — **ОШИБКА КОМПИЛЯЦИИ**. Итератор не поддерживает произвольный доступ.
    
- `list[5]` — **ОШИБКА КОМПИЛЯЦИИ**. У списка нет оператора индексирования.
    

Если вам нужно прыгнуть на 5 позиций вперед, вы обязаны использовать функцию `std::advance(it, 5)`, которая под капотом сделает цикл из пяти шагов (`++it`), что займет время $O(N)$.

### Главная суперспособность: `splice()`

Только у списка есть метод `splice` (сращивание). Это операция, которая позволяет за $O(1)$ перенести элементы из одного списка в другой **без копирования и без выделения новой памяти**.

Список просто "вырезает" цепочку узлов, переподключая указатели `next` и `prev`, и вставляет их в другое место.

``` cpp
#include <iostream>
#include <list>

int main() {
    std::list<int> list1 = {1, 2, 3};
    std::list<int> list2 = {10, 20, 30};

    auto it = list1.begin();
    std::advance(it, 1); // it указывает на '2'

    // Переносим ВЕСЬ list2 в list1 прямо перед итератором it
    // list2 после этого станет пустым!
    list1.splice(it, list2); 

    // list1 теперь: 1, 10, 20, 30, 2, 3
    for (int x : list1) std::cout << x << " "; 
    
    return 0;
}
```

### Темная сторона списка: Кэш-промахи (Cache Misses)

Будучи студентом факультета прикладной математики и информатики, вы должны знать главную причину, по которой создатель C++ Бьёрн Страуструп советует **избегать `std::list` по умолчанию** и всегда использовать `std::vector`.

Дело в архитектуре современных процессоров (CPU).

Когда процессор читает данные из оперативной памяти, он берет не один байт, а целую "кэш-линию" (обычно 64 байта).

- У **вектора** элементы лежат подряд. Читая `vec[0]`, процессор заодно загружает в свой сверхбыстрый кэш `vec[1]`, `vec[2]` и т.д. Обход вектора работает молниеносно (пространственная локальность).
    
- У **списка** каждый узел лежит в случайном месте памяти. Переход по указателю `next` почти всегда приводит к **кэш-промаху (cache miss)**. Процессору приходится простаивать сотни тактов, ожидая, пока данные подтянутся из медленной RAM.
    

В реальности, даже если вам нужно часто вставлять элементы в середину, `std::vector` часто оказывается быстрее `std::list`, потому что сдвиг непрерывного куска памяти (через `memmove`) работает быстрее, чем прыжки по указателям в кэше.

_(Стоит также упомянуть `std::forward_list` — это односвязный список, введенный в C++11. Он не имеет указателя `prev` и не знает своего размера, но он максимально экономно расходует память)._

``` cpp
template<typename T>
class DList {
private:
  struct Node {
    T value;
    Node* next;
    Node* prev;

    Node(const T& v = T(), Node* n = nullptr, Node* p = nullptr) : value(v), next(n), prev(p) {}
    ~Node() {}
  };

  Node* first;
  Node* last;

public:
  template<bool isConst>
  class common_iterator {
  private:
    std::conditional_t <isConst, const Node*, Node*> currentNode;
  public:
    common_iterator(Node* node = nullptr); currentNode(node) {}
    ~common_iterator() {}

    std::conditional_t <isConst, const Node*, Node*> getCurrentNode() const {
      return currentNode;
    }
    std::conditional_t <isConst, const T&, T&> getCurreoperator *() {
      return currentNode->value;
    }
    common_iterator& operator ++() {
      currentNode = currentNode->next;
      return *this;
    }
    std::conditional_t <isConst, const Node*, Node*> operator ->() {
      return currentNode;
    }
    bool operator != (const common_iterator& i) const {
      return (!(*this == i));
    }
    bool operator == (const common_iterator& i) const {
      return currentNode == i.currentNode;
    }
  };
  using iterator = common_iterator<false>;
  using const_iterator = common_iterator<true>;

  iterator begin() const {
    return iterator(first);
  }
  iterator end() const {
    if (last != nullptr) {
      return iterator(last->next);
    }
    return iterator(last);
  }
  
    DList() {
    first = nullptr;
    last = nullptr;
  }
  ~DList() {
    if (first == nullptr) {
      delete last;
      return;
    }
    if (last == nullptr) {
      delete first;
      return;
    }
    while (last->prev != nullptr) {
      Node* remove = last->prev;
      last->prev = remove->prev;
      delete remove;
    }
    delete last;
  }

  bool isEmpty() {
    return first == nullptr;
  }

  void push_back(const T& value) {
    if (first == nullptr) {
      Node* newNode = new Node(value);
      first = newNode;
      return;
    }
    if (last == nullptr) {
      Node* newNode = new Node(value);
      last = newNode;
      first->next = last;
      last->prev = first;
      return;
    }
    Node* curLast = last;
    Node* newNode = new Node(value);
    newNode->prev = curLast;
    curLast->next = newNode;
    last = newNode;
  }

  void pop_back() {
    if (first == nullptr) {
      return;
    }
    if (last = nullptr) {
      delete first;
      return;
    }
    Node* newLast = last->prev;
    newLast->next = nullptr;
    delete last;
    last = newLast;
  }

  void insert(iterator i, const T& value) {
    if (isEmpty()) {
      return;
    }
    
    Node* node = i.getCurrentNode();
    Node* newNode = new Node(value);
    Node* prevNode = node->prev;

    newNode->next = node;
    newNode->prev = prevNode;

    node->prev = newNode;

    if (prevNode != nullptr) {
      prevNode->next = newNode;
    }
    else {
      first = newNode;
    }
  }
  void reverse() {
    Node* node = first;

    while (node != nullptr) {
      Node* next = node->next;
      Node* newPrev = node->prev;
      node->prev = node->next;
      node->next = newPrev;

      node = next;
    }
    Node* newFirst = last;
    last = first;
    first = newFirst;
  }
};
```

# 12.7 Map

 **Map** (часто называют «словарем» или «ассоциативным массивом») — это структура данных, которая хранит элементы в виде пар **«ключ-значение»** (key-value).

В стандартной библиотеке C++ эта структура представлена контейнером `std::map`.

Основные особенности `std::map`:

1. **Уникальность ключей:** Каждый ключ может встречаться в коллекции только один раз. Вы не можете иметь два одинаковых ключа.
    
2. **Связь:** Каждому ключу соответствует ровно одно значение. Зная ключ, вы можете очень быстро найти, прочитать или изменить связанное с ним значение.
    
3. **Автоматическая сортировка:** В `std::map` элементы всегда хранятся в отсортированном виде по ключу. «Под капотом» это чаще всего реализуется с помощью структуры данных, которая называется красно-черным деревом.
    
4. **Сложность операций:** Поиск, добавление и удаление элементов происходят за логарифмическое время — $O(\log N)$.
    

**Аналогия из жизни:**

Обычный толковый словарь. Слово, значение которого вы ищете, — это **ключ** (он уникален, слова идут по алфавиту). Само определение слова — это **значение** (оно может быть любым, и у разных слов могут быть похожие определения).

**Небольшой пример на C++:**

Допустим, нужно сохранить оценки студентов. Имя (строка) будет ключом, а оценка (целое число) — значением.

C++

```
#include <iostream>
#include <map>
#include <string>

int main() {
    // Объявляем map, где ключ - строка, а значение - число
    std::map<std::string, int> grades;

    // Добавляем элементы
    grades["Иван"] = 5;
    grades["Анна"] = 4;
    
    // Можно добавить и так:
    grades.insert({"Петр", 3});

    // Обращение к значению по ключу
    std::cout << "Оценка Ивана: " << grades["Иван"] << std::endl;

    return 0;
}
```

Также в C++ существует `std::unordered_map`. Она делает то же самое, но не сортирует элементы по ключу. За счет использования хеш-таблиц вместо деревьев поиск и вставка в `unordered_map` работают быстрее — в среднем за $O(1)$.

**хеш-таблица** (hash table) — это структура данных, которая позволяет хранить пары «ключ-значение» и обеспечивает невероятно быстрый поиск, добавление и удаление элементов. Именно она лежит в основе `std::unordered_map` в C++.

Если обычный массив использует для доступа к элементам порядковые номера (0, 1, 2...), то хеш-таблица позволяет использовать в качестве «индекса» что угодно — строку, пользовательский класс или дробное число.

### Как это работает «под капотом»

Хеш-таблица состоит из двух главных компонентов:

1. **Массив (корзины или buckets):** Участок памяти, где физически хранятся данные.
    
2. **Хеш-функция:** Это математический алгоритм. Он берет ваш ключ (например, строку `"Иван"`) и превращает его в псевдослучайное целое число (хеш-код). Затем это число преобразуется в допустимый индекс массива (обычно через взятие остатка от деления на размер массива).
    

**Процесс записи:**

Допустим, вы хотите сохранить значение по ключу: `grades["Иван"] = 5`.

1. Ключ `"Иван"` передается в хеш-функцию.
    
2. Хеш-функция вычисляет, что для `"Иван"` индекс равен, например, `42`.
    
3. Значение `5` (вместе с самим ключом `"Иван"`) записывается в ячейку массива под номером `42`.
    

**Процесс чтения:**

Когда вы запрашиваете `grades["Иван"]`, хеш-функция снова мгновенно вычисляет индекс `42`, и программа сразу обращается к этой ячейке памяти. Ей не нужно перебирать другие элементы. Именно поэтому среднее время выполнения операций в хеш-таблице составляет $O(1)$.

### Проблема коллизий

Множество возможных ключей (например, всех возможных строк) бесконечно, а размер массива в памяти строго ограничен. Поэтому неизбежна ситуация, когда хеш-функция выдает **одинаковый индекс для двух совершенно разных ключей**. Это явление называется **коллизией**.

В C++ (в реализации `std::unordered_map`) эта проблема чаще всего решается **методом цепочек** (chaining):

- Каждая ячейка массива хранит не сам элемент, а указатель на связный список.
    
- Если для нового ключа вычисляется индекс, который уже занят другим ключом, новый элемент просто добавляется в конец списка в этой ячейке.
    
- При поиске элемента программа сначала мгновенно находит нужную ячейку массива, а затем линейно пробегается по короткому связному списку внутри нее, сверяя ключи на точное совпадение.
    

### Плюсы и минусы

**Преимущества:**

- Непревзойденная скорость в среднем случае — $O(1)$ для поиска, вставки и удаления.
    

**Недостатки:**

- **Худший случай:** Если хеш-функция подобрана неудачно и все ключи дают один и тот же индекс (сплошные коллизии), хеш-таблица вырождается в один длинный связный список. Время поиска падает до $O(N)$.
    
- **Отсутствие порядка:** Данные лежат хаотично. Вывести элементы по алфавиту не получится (в отличие от основанной на красно-черном дереве `std::map`).
    
- **Расход памяти:** Хеш-таблица должна иметь много пустых ячеек (низкий load factor), чтобы минимизировать шанс коллизий, поэтому она потребляет больше оперативной памяти.
